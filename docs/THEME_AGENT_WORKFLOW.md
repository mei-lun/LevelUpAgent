# 主题适配 Agent 工作流程

本文是 Agent 在 LevelUpAgent 中新建、迁移或审查主题时必须遵循的执行规程。开始工作前完整阅读 [THEME_DEVELOPMENT.md](./THEME_DEVELOPMENT.md) 和 [THEMES.md](./THEMES.md)。

## 一、输入和目标

开始前从用户请求和工作区确认：

- LevelUpAgent 仓库路径。
- 主题源仓库或参考图路径。
- 目标平台和需要交付的产物。
- 用户是否要求结构完全匹配、只换颜色，或两者兼有。
- 用户是否授权删除旧平台代码、构建安装包或修改宿主。

缺少非关键细节时优先从本地项目发现，不要把可检查的问题交还用户。

## 二、保护现有工作

1. 阅读两个仓库内的 `AGENTS.md` 和引用指令。
2. 运行 `git status --short`，记录已有修改和未跟踪文件。
3. 不覆盖、不回滚、不清理用户已有改动。
4. 搜索优先使用 `rg` 或 `rg --files`。
5. 文本修改使用 `apply_patch`；批量删除只有在用户明确授权时执行。

## 三、审计主题源

输出一份内部清单：

- 入口脚本、CSS、manifest、素材、测试和发布产物。
- 哪些文件属于旧宿主或旧平台。
- 主题真正使用的素材及其来源。
- 参考布局的区域、交互、响应式行为和窗口装饰。
- 主题是否依赖注入 JavaScript、远程 URL、绝对路径或私有状态。

不得直接搬运旧宿主注入器。只提取视觉语言、合法素材和可复用的设计规则。

## 四、审计 LevelUpAgent

至少检查：

- `docs/THEMES.md`
- `src-tauri/src/theme.rs`
- `src/lib/types.ts`
- `src/lib/bridge.ts`
- `src/lib/storage.ts`
- `src/App.tsx`
- `src/components/DeclarativeLayout.tsx`
- `layouts/default.layout.json`
- `docs/LAYOUTS.md`
- `src/App.css`
- `src-tauri/capabilities/default.json`

确认当前宿主已经支持哪些节点、插槽、数据路径与动作，不能在 `layout.json` 中猜测未注册能力。

## 五、做出适配决策

按顺序回答：

1. 只用作用域 CSS 能否完成？能则选择 `standard`，禁止改宿主。
2. 声明式布局节点、插槽、数据与动作能否满足？能则创建独立 `layout.json`，禁止改宿主。
3. 是否缺少必须公开的数据、动作或真实功能插槽？只有此时扩展宿主契约。

在 commentary 更新中向用户说明选择了哪一档，以及为什么需要或不需要宿主改动。

## 六、先完成独立主题包

实施顺序固定为：

1. 创建稳定 manifest ID 和版本。
2. 将必要素材集中到主题自己的 `levelup/assets/`。
3. 编写全部作用域化 CSS。
4. 编写构建器，将素材内嵌为 `data:` URL。
5. 编写包测试。
6. 生成 `.levelup-theme`。

在宿主改动前，主题包本身应尽可能完整。

## 七、最小化宿主扩展

如果必须扩展布局宿主能力：

1. 同步修改 Rust schema 校验、TypeScript 类型和前端运行时注册表。
2. 新能力必须能被多个主题复用，并且只公开安全、非敏感数据。
3. `App.css` 只定义中性结构能力和响应式规则，不放主题品牌颜色。
4. 具体视觉全部留在主题作用域 CSS。
5. 切回默认时恢复默认布局、CSS 和窗口装饰。
6. 原生窗口权限只添加实际调用项。
7. 新增 Rust 校验测试和必要的前端验证。

如果发现宿主改动正在变成某主题的复制品，应停止并重新抽象布局契约。

## 八、验证顺序

按以下顺序验证，前一步失败时先修复再继续：

1. 主题构建器运行成功。
2. 主题包自动测试通过。
3. `pnpm check`。
4. `pnpm build`。
5. `cargo test --manifest-path src-tauri/Cargo.toml`。
6. `pnpm tauri build --no-bundle`。
7. 启动真实桌面程序并截图比对参考布局。
8. 真实操作安装、切换、重启和卸载。
9. 若有自定义标题栏，真实点击最小化、最大化、还原和关闭。
10. 最终执行 `pnpm tauri build --bundles nsis`。

Web 预览不能替代真实 Tauri 窗口验证。

## 九、视觉验收方法

将参考图和实机截图按区域逐项比较：

| 区域 | 检查内容 |
| --- | --- |
| 标题栏 | 是否重复、标题位置、拖动、窗口按钮位置与功能 |
| 工具栏 | 高度、图标、选中态、按钮实际动作 |
| 左栏 | 宽度、项目树层级、搜索和底部状态 |
| 中央 | 会话宽度、消息间距、代码块、审批和 Diff |
| 输入区 | 标题、附件、文本区伸展、工具栏贴底、发送/停止 |
| 右栏 | 标签页、资料、分组、滚动和窄屏降级 |
| 状态栏 | 高度、状态信息、权限显示 |

不得只凭“颜色相似”宣称完整适配。

## 十、禁止事项

- 不允许主题执行 JavaScript。
- 不允许远程 CSS、远程图片或 `@import`。
- 不允许未作用域的全局 CSS。
- 不允许隐藏审批、权限或安全提示来追求视觉一致。
- 不允许永久改变默认主题窗口装饰。
- 不允许把窗口按钮做成无功能装饰。
- 不允许依赖主题源仓库的绝对路径。
- 不允许在没有用户授权时删除旧项目或发布外部内容。
- 不允许为了主题修改 Provider、API Key、数据库或 Agent 执行逻辑。

## 十一、失败处理

- 构建失败：先保留完整错误，定位是主题包、前端、Rust、工具链还是文件锁。
- EXE 被占用：关闭正在运行的测试程序后重试，不删除构建缓存。
- 主题不能安装：检查扩展名、JSON、大小、作用域、远程构造、companion 文件名和布局 schema。
- 按钮不响应：检查拖动区域是否覆盖按钮，再检查 Tauri 权限。
- 切回默认仍残留：检查活动 CSS 清空、结构节点条件渲染和窗口装饰恢复。
- 参考资产缺失：停止复制，向用户报告缺失或使用已授权替代素材。

## 十二、交付格式

最终回复必须包含：

- 完成了哪些主题能力。
- 是否修改 LevelUpAgent，以及修改原因。
- 主题包绝对路径和可点击链接。
- 安装包绝对路径和可点击链接（如果用户要求构建）。
- 执行过的测试及结果。
- SHA-256（发布产物需要）。
- 已知限制，例如未签名安装包。

不要把临时截图脚本、测试进程或 QA 图片留在仓库中。
